# 开发日志 2026-06-27

## 六、后端服务化及面向对象重构 (Phase 4)

**时间戳：2026-06-27 19:10**

- **问题背景**：原有的项目偏向于一次性的脚本乘算。根据最新需求（`后端设计逻辑.md`），系统需升级为支持多实体独立运作的 Serverless/Web API 架构，能够在一个“战斗空间”中模拟干员和敌人的交互。
- **实施细节**：
  1. **目录结构物理重组**：彻底清理了旧版的 `calculator` 等静态脚本；`data/` 目录细分为 `Operators`, `Enemies`, `raw_data`, `parsed_data`。
  2. **批量自动获取工具**：在 `crawler` 新增了 `batch_fetch.py`，其可以通过请求 GitHub (ArknightsGameData) 拉取游戏全量的 6 星干员（共 182 名），并接入已有的 `llm_cleaner.py` 组成全自动清洗流水线。
  3. **面向对象的架构铺垫**：
     - 构建了 `backend/function/logic/base_operator.py`（所有干员的基础抽象类）。
     - 构建了独立的 `backend/function/battle/enemy.py`（敌人状态机模型）。
     - 部署了核心的 `backend/function/battle/engine.py` (宏观战斗公式演算器：`BattleEnvironment`)。
  4. **动态构建器 (`builder.py`)**：为了满足“读取干员后动态搭建不同职业父类和子类”的需求，开发了扫描器 `builder.py`，用于动态在 `professions.py` 追加基类代码，并自动在 `operators/` 目录下吐出继承体系。
  5. **API 层落地**：在 `backend` 完成 `main.py` 的重写，引入了 `FastAPI` 构建标准的 REST 服务 `/api/simulate`。

- 20:15 修复了批量爬虫中的 NoneType 报错，并清理了无效的 json 文件。
- 20:15 运行 builder.py 动态生成了 130 位干员的 Python 对象类骨架以及相关的职业父类。
