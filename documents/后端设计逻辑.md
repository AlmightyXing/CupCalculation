# backend设计逻辑

## 运行流程总览（执行顺序）

1. 使用脚本在PRTS网站对干员进行数据爬取，获得包含网络格式的raw文本数据
2. 使用LLM模型/清洗脚本对数据进行清洗，获得结构化的纯净json信息文本
3. agent根据爬取到的信息文本完成干员技能逻辑的代码实现，形成一个个独立的对象
4. agent将敌人数据进行代码实现，根据不同属性组合创建独立对象
5. agent搭建战斗环境，对选择进行模拟战斗的干员和敌人对象进行调用，在战斗环境中进行代码层面的逻辑运行并得出伤害数据

## 前后端交互逻辑

- 前端可以选择干员和对应的敌人配置，作为输入发送给后端
- 后端接受输入，搭建作战空间，调用对应干员与敌人对象
- 将选中的敌人和干员对象加入对战空间
- 运行战斗空间的代码进行模拟并得出伤害数据（包括总伤、DPS、伤害构成等）
- 将模拟结果返回给前端
- 前端进行展示（该内容在前端设计中进行详细说明）

## 后端架构说明

- 后端采用Serverless架构，所有干员技能逻辑均为独立的Function，战斗空间也是独立的Function（测试期间所有Function保存在本地）
- 干员信息（攻击力、防御、生命等）以及敌人信息储存在Document-Oriented数据库中（测试期间均保存在本地）

### 项目结构

- backend
    - crawler（用于爬取PRTS网站数据）
    - parser（用于清洗数据）
    - logic（用于实现干员技能逻辑）
    - battle（用于搭建战斗环境）
    - main（用于启动后端）
- data
    - Operaters
    - Enemies
    - raw_data（用于储存爬取到的原始数据）
    - parsed_data（由于储存清洗后的结构化数据）
- frontend
- logs（用于记录开发日志和运行时日志，开发日志格式为`log-YYMMDD.md`，每次更新日志时需要在更新内容前加上精确到分钟的时间戳）
- documents（用于储存PRD、前后端设计逻辑、原型文件等项目相关文档图片）

### 对象函数设计

- 不要在一开始就定义全部的基础抽象类。在所有6星干员的信息爬取完毕后，再开始干员对象的逻辑搭建。每读取一个干员的信息，根据其职业是否为已有父类，进行动态搭建。
- 对象按照干员的职业划分为多个父类，父类下的干员共享相同的特性、攻击间隔、再部署时间等基础属性
    例：所有**重射手**的攻击间隔为1.6s，阻挡数为1，再部署时间为70s。
- 在进行程序设计时，如果遇到了未见过的职业，需要先建立该职业的父类，在父类中定义该职业独有的特性、攻击间隔、再部署时间等基础属性，然后将该干员作为子类继承该父类。后续所有相同该职业的干员都可以直接使用该父类。
- 干员的职业为特性中描述的职业，如**重射手**、**神射手**、**领主**等，而非狙击、术师、近卫等大类。
